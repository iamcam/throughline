# Podcast Knowledge Engine — Future Scope

> Options for expanding capability after v1.0. Each section is independent.
> Includes rationale, estimated effort, and interview story value.

---

## Tier 1 — High Value, Manageable Scope

### 1.1 Graph RAG

**What it is:** Build a knowledge graph over podcast content alongside (not instead of) the flat vector index. Entities (people, companies, ideas, events) become nodes; relationships between them become edges. Retrieval traverses the graph in addition to doing vector search.

**Why it matters:** Standard RAG retrieves isolated chunks. Graph RAG can answer "how has the host's view on X evolved across episodes?" or "what's the connection between topic A and person B?" — questions that require connecting dots across the corpus.

**Implementation path:**
- Post-chunking: run NER (spaCy or LLM-based) over chunks → extract entities
- Build entity and relationship tables in Postgres
- Add graph traversal to retrieval: vector search finds entry points, graph traversal expands context
- LightRAG is an open-source option if you want a library; building it yourself is the better story

**Effort:** 2-3 weekends
**Interview story:** "I started with flat RAG, measured where it fell short on multi-episode questions, and added a graph layer. Here's the schema and here's an example where graph traversal found a connection vector search missed."

---

### 1.2 Persistent Conversations

**What it is:** Save chat sessions to the DB. Browse and resume past conversations.

**Why it matters:** Turns the tool from a query interface into a research companion. You can build up a thread of inquiry over multiple sessions.

**Implementation path:**
- Add `conversations` and `conversation_messages` tables
- Session create/resume endpoints
- Frontend: conversation list in sidebar, click to restore
- Optional: conversation title auto-generated by LLM from first exchange

**Effort:** 1 weekend
**Interview story:** Low architectural novelty, but shows product thinking — "I realized ephemeral sessions limited how people actually use a research tool."

---

### 1.3 Query Rewriting

**What it is:** Before retrieval, have the LLM rewrite the user's query using conversation context to make it more retrievable.

**Example:**
User: "What does he think about that?"
Context: last message was about Lex discussing consciousness
Rewritten query: "Lex Fridman views on consciousness and AI"

**Why it matters:** Pronoun resolution and context-dependent queries are a known RAG failure mode. This is a well-understood fix and shows you know the failure modes.

**Implementation path:**
- Pre-retrieval step in `query/engine.py`
- LLM call: "Given this conversation, rewrite the user's query for semantic search"
- Use rewritten query for retrieval, original for response
- Log both in telemetry — makes the improvement measurable

**Effort:** 1 day
**Interview story:** "I noticed retrieval quality dropped on follow-up questions. I added query rewriting and here's the before/after in the traces."

---

### 1.4 Multi-Feed Persona Synthesis

**What it is:** Cross-feed queries. "What would Lex and Dwarkesh each say about superintelligence?" Returns synthesized perspectives from multiple hosts across multiple feeds.

**Why it matters:** The persona-query angle is the most memorable part of this product. This makes it genuinely unique.

**Implementation path:**
- `search_knowledge_base` tool already supports `feed_id` filter — extend to accept a list
- System prompt instructs LLM to attribute answers to specific hosts
- Frontend: multi-feed scope selector in chat
- Optional: "debate mode" — synthesize opposing views side by side

**Effort:** 1 weekend (mostly frontend + prompt engineering)
**Interview story:** "The cross-host comparison is the killer feature. Here's a query where two hosts directly contradict each other — the system surfaces both with citations."


### 1.5 Speaker Diarization + Full Speaker Identity

**What it is:** Determine who is speaking at each moment in an episode, producing real `SPEAKER_00`, `SPEAKER_01`, etc. labels. This is the prerequisite for meaningful multi-speaker speaker filtering in chat queries ("what does the guest think about X?").

**Why deferred from v1:** CPU/local diarization via Pyannote is impractical on non-CUDA hardware — 2–4× real-time means a 2-hour episode takes 4–8 hours to diarize on CPU. v1 uses `UNKNOWN` as the speaker sentinel for all segments. `SpeakerResolver` attempts to infer a single host name from the intro, which covers single-host podcasts adequately. Multi-speaker attribution waits for this work.

**What changes when this lands:**

*Schema:* No changes — `speaker_id` column already exists in `transcript_segments` and `chunks`. The `UNKNOWN` → `SPEAKER_XX` promotion is a data migration for existing episodes, not a schema change.

*Pipeline:* `PENDING_NAMES` status reinstated. After diarization assigns real speaker IDs, the pipeline pauses for user confirmation of speaker names. `SpeakerResolver` runs per diarized speaker rather than once for the whole episode.

*`SpeakerResolver`:* Return type changes back to `dict[str, InferredSpeaker | None]` — one entry per diarized speaker. The single-speaker path from v1 becomes a special case.

*Frontend:* Speaker naming page becomes a pipeline gate again for multi-speaker episodes.

**Implementation backends (choose one to start):**

- **Local Pyannote (CUDA required):** `pyannote/speaker-diarization-3.1`. Fast on GPU, impractical on CPU. Requires `HUGGINGFACE_TOKEN`. Already stubbed in `src/transcription/local.py`.
- **OpenAI Whisper API with diarization:** Single API call returns both transcript and speaker turns. Simplest path if already using OpenAI for LLM. Implement in `RemoteTranscriptionService`.
- **Generic remote sidecar:** Docker container running Whisper + Pyannote on GPU. HTTP contract mirrors existing `TRANSCRIPTION_SERVICE_URL` pattern. Good for self-hosted GPU.
- **Cloud GPU providers (RunPod, Modal):** POST audio, receive `TranscriptResult`-compatible JSON. One-weekend implementation per provider.

**Configuration when implemented:**
```
DIARIZATION_BACKEND=local|remote
DIARIZATION_MODEL=pyannote/speaker-diarization-3.1   # local backend
# Remote backend uses TRANSCRIPTION_SERVICE_URL
```

**Effort:** 1–2 weekends depending on backend choice. OpenAI API path is fastest.

**Interview story:** "I deferred diarization because Pyannote on CPU was a 4-hour wall on a 2-hour episode. I used `UNKNOWN` as an honest sentinel rather than `SPEAKER_00` — that distinction matters when you want to identify which old episodes need re-ingesting after diarization lands. Here's the re-ingestion path."

## Tier 2 — Meaningful Additions, Higher Effort

### 2.1 Automatic Feed Polling

**What it is:** Background scheduler checks subscribed feeds on a configurable interval and automatically ingests new episodes.

**Implementation path:**
- Add APScheduler or Celery Beat
- Configurable poll interval per feed
- Notification in frontend when new episode ready
- Requires thinking about queue management for concurrent ingestion

**Effort:** 1 weekend
**Note:** Mostly infrastructure work, low AI novelty. Good for product completeness, not for interview stories.

---

### 2.2 Episode Summarization and Show Notes

**What it is:** Auto-generate structured show notes per episode post-ingestion: key topics, key quotes, timestamps, people mentioned.

**Implementation path:**
- Post-ingestion step: LLM over full transcript (or hierarchical summary of chunks)
- Store in `episode_summaries` table
- Surface in frontend: episode detail page
- Export as markdown

**Effort:** 1 weekend
**Interview story:** "I built a two-level summarization pipeline — chunk-level summaries first, then episode-level synthesis — because the full transcript exceeded context window limits."

---

### 2.3 Semantic Search UI

**What it is:** Search mode separate from chat. Type a query, get ranked episode moments with timestamps — more like a search engine than a conversation.

**Why it matters:** Some questions don't need synthesis — you just want to find where something was said.

**Implementation path:**
- New frontend view: search input, results list with timestamp links
- Direct retriever call without LLM synthesis
- Highlight matched text in results

**Effort:** 1 weekend (mostly frontend)

---

### 2.4 Alternative Vector Backends

**What it is:** Implement the `VectorStore` protocol for Qdrant or Pinecone alongside pgvector.

**Why it matters:** Shows architectural thinking — you designed for this from day one. In interviews, "here's the interface, here's how I swapped implementations" is a clean story.

**Effort:** 1 day per implementation
**Note:** Only worth doing if you have a reason (e.g. pgvector performance limitations at scale, or you want to show Pinecone specifically).

## Tier 3 — Research / Experimental

### 3.1 Speaker Embedding and Voice Fingerprinting

**What it is:** Use audio-based speaker embeddings (pyannote speaker embeddings, or similar) to identify the same speaker across episodes — even if they were assigned different names. Useful for guest tracking across a long-running podcast.

**Why it matters:** "Speaker 1" in episode 3 might be the same person as "Speaker 2" in episode 7. Connecting those is genuinely hard and interesting.

**Effort:** 2-3 weekends, research-heavy

---

### 3.2 Temporal Reasoning — "How Has Their View Changed?"

**What it is:** Time-aware queries. "How has the host's position on AI safety evolved between 2022 and 2024 episodes?"

**Implementation path:**
- Episode date already stored — use it as a filter and in chunk metadata
- Retrieval: two passes — early episodes, recent episodes
- Synthesis: LLM compares and contrasts with explicit temporal framing

**Effort:** 1 weekend (mostly prompt engineering + retrieval strategy)
**Interview story:** High novelty. Most RAG demos are static. Temporal reasoning shows you thought about the unique properties of podcast content.

---

### 3.3 Confidence Scoring and Hallucination Detection

**What it is:** For each claim in an LLM response, assess whether it's grounded in the retrieved context or fabricated.

**Implementation path:**
- Post-synthesis step: structured prompt asks LLM to rate groundedness of each sentence
- Surface low-confidence claims in UI with a warning indicator
- Log in telemetry — makes hallucination rate measurable over time

**Effort:** 1 weekend
**Interview story:** "I added a groundedness check because the persona-query feature has real hallucination risk. Here's how I measure and surface it."

---

## What to Build Next (Recommended Order)

If you finish v1 and want to keep going, this is the highest-value sequence:

1. **Query rewriting** — 1 day, directly improves quality, measurable in Phoenix
2. **Episode summarization** — shows a two-level LLM pipeline, good interview story
3. **Temporal reasoning** — unique angle, memorable demo
4. **Graph RAG** — the "big" upgrade, strongest architectural story
5. **Multi-feed persona synthesis** — the killer demo feature

Graph RAG is deliberately not first — you'll understand your retrieval failure modes better after a few weeks of real use, which makes the graph design decisions more grounded.
